六种基本形状

反向代理

入口服务端侧

User → Proxy → App

代理代表后端服务接收请求。常做 TLS、路由、压缩、缓存、鉴权、负载均衡。

正向代理

出口客户端侧

App → Proxy → Internet

代理代表客户端访问外部目标。常用于企业出网、审计、代理下载、HTTP CONNECT。

SSH 本地转发

临时访问点对点

Local:15432 → SSH → DB:5432

把本机端口映射到 SSH 服务器可达的目标。适合临时访问远端内网数据库。

SSH 动态转发

SOCKS5按目标转发

Browser → socks5://127.0.0.1:1080 → SSH

SSH 在本机开一个 SOCKS 代理，由客户端每次请求时指定目标。

内网穿透

无公网 IP反向连接

Origin → Relay ← User

内网 agent 主动连公网 relay，用户再连 relay。适合 NAT 后服务暴露。

Overlay/VPN

私网互联路由

Node A ⇄ Virtual Network ⇄ Node B

给机器分配虚拟网段地址，用 ACL 和路由控制互访。适合长期私网访问。

最小判断流程

  1. 谁是用户？ 公网用户、公司内用户、服务进程、你本机？
  2. 谁能主动发起连接？ 公网能入站？只能出站？两边都在 NAT 后？
  3. 你要暴露服务还是访问资源？ 暴露服务偏反向代理/穿透；访问资源偏正向代理/隧道/VPN。
  4. 是临时还是长期？ 临时调试用 SSH 转发；长期生产用反代、网关、VPN/Overlay、零信任。
  5. 安全边界在哪里？ TLS 在哪里终止？谁鉴权？谁能直接打到源站？日志在哪里？

常用命令骨架

# 反向代理：Caddy 自动 HTTPS 到本机应用
caddy reverse-proxy --from example.com --to :9000

# SSH 本地转发：本机 15432 访问远端内网 DB:5432
ssh -L 127.0.0.1:15432:db.internal:5432 user@jump-host

# SSH 动态转发：本机开 SOCKS5 代理
ssh -D 127.0.0.1:1080 user@jump-host

# SSH 远程转发：把内网服务反向挂到公网服务器
ssh -R 127.0.0.1:18080:127.0.0.1:8080 user@public-server

# Docker 发布端口：宿主机 8080 → 容器 80
# 生产环境要明确绑定地址，避免默认暴露到所有网卡
docker run -p 127.0.0.1:8080:80 nginx
