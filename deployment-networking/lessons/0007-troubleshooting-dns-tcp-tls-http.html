Lesson 0007 · Troubleshooting

部署网络排障：DNS → TCP → TLS → HTTP → 代理 → 应用

本课练一个排障顺序：不要一上来改配置。先定位断层，再动手。

这一课的胜利标准

遇到“网站打不开 / 502 / 证书错误 / 后端没收到请求”时，你能按层次问问题，并用最小命令确认流量停在哪。

排障六层

DNS name ↓ TCP connection ↓ TLS handshake ↓ HTTP request/response ↓ Proxy routing ↓ Application behavior

大部分部署问题都能放进这条链。顺序很重要：DNS 错了，后面的 TLS/HTTP 都是在错误机器上发生；TCP 不通，就别先看应用日志。

每层问什么

层问题常用命令典型现象
DNS域名解析到哪？是否 CNAME？TTL 多久？dig app.example.com A解析到旧 IP、内外网解析不同
TCP目标 IP:port 能建立连接吗？curl -v telnet://host:443timeout、connection refused
TLS证书域名、SNI、CA、协议是否匹配？curl -v https://host/certificate verify failed、wrong version
HTTPHost、路径、状态码、重定向是否正确？curl -v -I https://host/path404、301 循环、401/403
代理反代是否路由到正确 upstream？头是否正确？代理日志 + curl --resolve502/504、后端 IP 错、Host 错
应用应用是否监听？是否信任代理头？ss -ltnp + 应用日志只监听容器内、生成 http URL

最小工具链

# 1. DNS
dig app.example.com A +short
dig app.example.com CNAME

# 2-4. TCP/TLS/HTTP 一起看
curl -v https://app.example.com/

# 5. 不改 DNS，指定某台机器验证反代
curl -v --resolve app.example.com:443:203.0.113.10 https://app.example.com/

# 6. 在服务器上看监听
ss -ltnp

# 7. 怀疑包根本没到服务器时抓包
sudo tcpdump -i any host 203.0.113.10 and tcp port 443

502 的定位例子

User → Proxy → App ↑ 502
  1. 用户能连到代理：否则不会得到代理发出的 502。
  2. 问题通常在代理到 upstream：后端没监听、端口错、协议错、Host/SNI 错、后端超时。
  3. 在代理机器上直接请求 upstream：curl -v http://127.0.0.1:3000/health。
  4. 看监听：ss -ltnp 'sport = :3000'。
  5. 如果 upstream 是 HTTPS，检查代理到后端的 SNI/证书信任。

证书错误的定位例子

  1. curl -v https://app.example.com/ 看证书 Subject/SAN、issuer、SNI。
  2. 确认 DNS 是否指向正确代理 IP。
  3. 用 --resolve 指向候选机器，确认哪台机器证书正确。
  4. 如果走 CDN/Cloudflare，再区分“用户到边缘证书”和“边缘到源站证书”。

不要做的事

  • 不要看到 502 就重启所有服务；先确认 upstream 是否可达。
  • 不要看到证书错误就盲目重签；先确认请求是不是到了正确机器。
  • 不要用浏览器缓存结果判断当前状态；排障优先 curl -v。
  • 不要忘记容器端口映射：容器内监听不等于宿主机可访问。
