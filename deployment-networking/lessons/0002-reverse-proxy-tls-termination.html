Lesson 0002 · Reverse Proxy

反向代理、TLS 终止与真实客户端信息

本课练一个部署动作：把公网 HTTPS 入口放在代理上，后端应用只监听本机或内网，并把必要的客户端信息传给后端。

这一课的胜利标准

你能解释 User → HTTPS → Proxy → App 里 TLS 在哪里结束、后端为什么需要 X-Forwarded-* 头，以及为什么后端服务不该裸露公网。

配套速查：部署网络命令速查

反向代理的最小路径

Browser │ HTTPS :443 ▼ Caddy / Nginx / HAProxy │ HTTP :3000 or HTTPS :3000 ▼ App process

反向代理站在服务端前面。用户以为自己访问的是应用，实际上先到代理；代理再请求后端应用。NGINX 官方文档把这个过程描述为：代理接收请求、发送给上游、取回响应、再返回给客户端。

TLS 终止在哪里？

代理终止 TLS

User ==HTTPS==> Proxy --HTTP--> App

最常见。证书在代理上，后端应用可以只跑 HTTP。适合单机或可信内网。

端到端 TLS

User ==HTTPS==> Proxy ==HTTPS==> App

后端也启 TLS。适合跨不可信网络、合规要求高、零信任环境。

“TLS 终止”不是坏事。关键是终止后到后端的那段网络是否可信，以及认证/授权是否仍然完整。

配置骨架

Caddy：最小 HTTPS 反代

app.example.com {
  reverse_proxy 127.0.0.1:3000
}

Caddy 知道域名后会默认自动启用 HTTPS、签发/续期证书，并把 HTTP 重定向到 HTTPS。

Nginx：保留真实请求信息

location / {
  proxy_pass http://127.0.0.1:3000;
  proxy_set_header Host $host;
  proxy_set_header X-Real-IP $remote_addr;
  proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  proxy_set_header X-Forwarded-Proto $scheme;
}

后端看到的直接来源通常是代理 IP，不是用户 IP。应用如果要生成绝对 URL、记录客户端 IP、判断是否 HTTPS，通常要读取这些代理头，并且只信任来自可信代理的头。

部署时的三个边界

后端监听地址

单机部署时优先让应用监听 127.0.0.1:3000，只让代理暴露公网。

源站绕过

如果后端端口公网可达，攻击者可以绕过代理层的 TLS、限流、WAF 或认证。

代理头信任

X-Forwarded-For 可被客户端伪造。应用只能信任由可信反代追加/覆盖后的值。

快速验证

# 看 TLS、连接 IP、请求/响应头
curl -v https://app.example.com/

# 绕过 DNS 指向某台新机器，但保持 Host/SNI
curl -v --resolve app.example.com:443:203.0.113.10 https://app.example.com/

# 在服务器上确认后端只监听本机
ss -ltnp 'sport = :3000'

练习

场景

单机部署一个 API，Caddy 负责公网 HTTPS，Node.js 后端应该优先监听哪里？

场景

后端生成回调 URL 时总生成 http://，但用户访问入口是 HTTPS。优先检查哪个代理头？
