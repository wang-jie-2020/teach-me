Lesson 0005 · Tunneling

内网穿透：没有公网 IP 时如何暴露服务

本课练一个判断：当源站不能被公网主动连接时，就让源站主动连出去，建立一条可反向承载流量的通道。

这一课的胜利标准

你能画出 Origin → Relay ← User，并知道它和反向代理、SSH -R、VPN 的区别。

为什么需要内网穿透

很多服务位于 NAT、防火墙、家宽、公司内网后面：公网用户无法直接向它发起连接。内网穿透的思路是反过来：内网机器运行一个 agent，主动向公网 relay 建立出站连接；用户访问 relay，relay 再把流量送回内网 agent。

Home / Office Origin │ outbound-only connection ▼ Public Relay / Edge ▲ │ user request User

Cloudflare Tunnel 官方文档明确描述了这种 outbound-only 模型：cloudflared 从源站主动连到 Cloudflare 网络，源站不需要公网可路由 IP。

三种常见实现

SSH -R

临时小范围

ssh -N -R 127.0.0.1:18080:127.0.0.1:8080 user@public-server

远端服务器监听端口，流量通过 SSH 回到本地服务。

frp / ngrok 类

自建 relay多协议

内网跑 client，公网跑 server。适合自建穿透、映射 TCP/HTTP 服务。

Cloudflare Tunnel

托管边缘身份策略

源站跑 cloudflared，通过 Cloudflare 边缘暴露 HTTP、SSH 等资源，可叠加 Access 身份策略。

它和反向代理的关系

很多内网穿透产品的公网侧仍然表现得像反向代理：用户访问域名，边缘节点转发到后端。区别在于后端连接方向：

普通反向代理

Proxy → Origin

代理能主动连到源站。源站通常有公网 IP、内网路由或防火墙放行。

内网穿透

Origin → Relay

源站主动连 relay。relay 不需要能直接拨入源站。

安全边界

  • 穿透不是“自动安全”。它只是改变连接方向，服务仍然可能暴露给公网。
  • 优先叠加身份认证：Access、Basic Auth、OIDC、mTLS、应用登录，不要只靠“隐藏 URL”。
  • 限制暴露范围：只映射必要端口和路径；管理后台不要默认暴露给所有人。
  • 确认源站是否还能被其他路径直连。如果能，需要防火墙或安全组封住绕过路径。

选型判断

场景优先考虑原因
临时演示本机服务SSH -R / ngrok快，用完关
家中 NAS 长期访问Cloudflare Tunnel / Tailscale少开入站端口，可加身份控制
自有 VPS + 多内网服务frp自控 relay，多协议映射
团队访问多个内网网段Overlay/VPN不是暴露单个服务，而是建立受控私网互访
