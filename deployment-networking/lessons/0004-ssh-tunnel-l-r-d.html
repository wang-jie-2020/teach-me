Lesson 0004 · SSH Tunnel

SSH Tunnel：-L、-R、-D 与跳板机

本课练一个动作：看到 SSH 转发命令时，能说出“监听端口在哪里、最终连接从哪里发起、目标是谁”。

这一课的胜利标准

你能根据场景选择 -L 本地转发、-R 远程转发、-D 动态转发，并知道绑定到 127.0.0.1 和 0.0.0.0 的安全差异。

先看监听端口在哪

-L 本地转发

Local listens 127.0.0.1:15432 → SSH → db.internal:5432

本机开端口。适合“我从本机访问远端内网资源”。

-R 远程转发

Remote listens public:18080 → SSH → local:8080

远端 SSH 服务器开端口。适合“把我这边服务挂到远端入口”。

-D 动态转发

Local SOCKS listens 127.0.0.1:1080 → SSH → chosen target

本机开 SOCKS 代理。每个请求的目标由浏览器/curl 决定。

命令拆解

# -L [本机绑定地址:]本机端口:目标主机:目标端口 SSH服务器
ssh -N -L 127.0.0.1:15432:db.internal:5432 user@jump-host

# 连接 db 的动作由 jump-host 发起，所以 db.internal 只要 jump-host 能解析/访问即可。

# -R [远端绑定地址:]远端端口:本地目标主机:本地目标端口 SSH服务器
ssh -N -R 127.0.0.1:18080:127.0.0.1:8080 user@public-server

# -D [本机绑定地址:]本机端口 SSH服务器
ssh -N -D 127.0.0.1:1080 user@jump-host

跳板机不是端口转发，但经常搭配

# 经 bastion 登录目标机器
ssh -J user@bastion user@target.internal

# 经 bastion 建立本地转发到目标内网数据库
ssh -J user@bastion -N -L 127.0.0.1:15432:db.internal:5432 user@target.internal

-J 解决“怎么连到 SSH 服务器”；-L/-R/-D 解决“连接后转发什么流量”。

安全边界

  • 优先绑定 127.0.0.1。绑定 0.0.0.0 意味着同网段/公网可能访问你的转发端口。
  • 服务端可用 AllowTcpForwarding 限制是否允许本地/远程转发。
  • SSH 隧道适合临时运维和调试。长期生产入口应考虑反代、VPN/Overlay、零信任访问，并有审计。
  • 不要把数据库等敏感服务通过远程转发直接暴露给公网。

练习

场景 1

你要让本机数据库客户端连到远端内网的 PostgreSQL，应该优先用哪个？

场景 2

你想让浏览器临时访问多个远端内网页面，不想为每个端口写一条转发。选哪个？
