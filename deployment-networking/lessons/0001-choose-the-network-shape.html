Lesson 0001 · Deployment Networking

先判断流量形状，再选网络工具

本课只练一个动作：看到部署网络需求时，先判断“谁访问谁、连接由谁发起”，再决定用反向代理、正向代理、隧道、内网穿透还是 VPN/Overlay。

这一课的胜利标准

读完后，你应该能把一个需求归入正确的“网络形状”，而不是被工具名牵着走。

配套速查：部署网络手段速查图

核心模型：问两个问题

1. 谁访问谁？

公网用户访问你的应用？你的服务访问外网？你本机访问远端内网资源？多台机器长期互访？

2. 谁能主动连接？

公网能直接连到服务端吗？内网机器只能出站吗？两边是否都在 NAT 或防火墙后面？

这两个问题比“我要不要用 Nginx / frp / SOCKS5”更先出现。工具只是网络形状的实现。

五个常见形状

公网用户 → 你的应用

User → Edge Proxy → App

反向代理负载均衡Ingress

代理站在服务端前面，代表应用接收请求。常做 TLS 终止、路由、鉴权、缓存、限流、健康检查。

你的服务 → 外部目标

App → Forward Proxy → Internet

正向代理CONNECT出口网关

代理站在客户端/服务这边，代表它访问外部目标。常用于受控出网、审计、统一出口 IP。

你本机 → 远端内网资源

Laptop:15432 → SSH → db.internal:5432

SSH -L跳板机临时调试

本机开一个端口，经 SSH 安全通道转到远端内网服务。适合临时访问数据库、管理后台。

内网服务 → 公网入口

Origin → Tunnel/Relay ← User

内网穿透SSH -RCloudflare Tunnel

内网机器主动连公网 relay，然后外部用户连 relay。适合没有公网 IP、入站端口不能开放的场景。

多网络节点长期互访

Node A ⇄ Overlay/VPN ⇄ Node B

VPNOverlaySubnet Router

给设备建立一张虚拟网络，靠路由和 ACL 控制互访。适合团队长期访问内网、跨云互联。

一个反例：同一个 Nginx，不同角色

Nginx 常被当作反向代理：公网用户访问 https://app.example.com，Nginx 再转发到 127.0.0.1:3000。

app.example.com {
  reverse_proxy 127.0.0.1:3000
}

但“用了 Nginx”并不自动说明网络形状。你要看它站在哪一侧：站在服务端前面，就是反向代理；站在客户端出口侧，就是正向代理/出口代理。

练习：先选形状

场景 1

你有一个 Node.js 服务监听 127.0.0.1:3000，希望用户通过 https://api.example.com 访问，并自动续期证书。

场景 2

你在本机调试，需要临时访问只能从跳板机看到的 db.internal:5432，不想改数据库防火墙。

场景 3

家里 NAS 后面是运营商 NAT，没有公网 IP；你希望外面通过一个域名访问它的 Web 管理页。

安全边界：每次都问这四句

  1. 这个方案有没有把服务暴露到公网？
  2. TLS 在哪里终止？终止后到后端是否仍加密？
  3. 认证发生在哪里？应用层、代理层、身份代理，还是根本没有？
  4. 源站是否还能被绕过代理直接访问？

推荐主源

先读 MDN 的 Proxy servers and tunneling。它用最少篇幅区分 forward proxy 与 reverse proxy，并指出反向代理常用于负载均衡、认证、解密和缓存。

本课还参考：NGINX Reverse Proxy、OpenSSH ssh(1)、Cloudflare Tunnel、Tailscale Subnet Routers。
