类型收窄 Narrowing

本课小胜利：拿到一个联合类型（union type）后，能用运行时检查让 TypeScript 准确知道“此刻是哪一种类型”，并用可辨识联合 + 穷尽检查写出不漏分支的业务逻辑。约 28 分钟。

1 · 为什么需要收窄：联合类型只能使用“共同能力”

上一课学过联合类型：number | string 表示一个值可能是数字，也可能是字符串。问题是，编译器在不知道具体是哪一种之前，只允许你使用每个成员都安全支持的操作。

function format(value: number | string): string {
  return value.toUpperCase();
  //     ❌ number 上不存在 toUpperCase
}

即使你心里知道调用时“通常传字符串”，函数体仍必须对类型声明里的所有可能负责。正确做法是先问一个运行时能回答的问题，再使用对应类型的能力：

function format(value: number | string): string {
  if (typeof value === "string") {
    return value.toUpperCase(); // 此处 value: string
  }
  return value.toFixed(2);    // 此处 value: number
}

这种让类型从较宽的联合类型缩小为更具体类型的过程，叫类型收窄（narrowing）；触发收窄的检查叫类型守卫（type guard）。

编译器做的事：控制流分析

TypeScript 不只是逐行看类型，它会分析 if、switch、return 等控制流。上例的第一个分支已经处理并返回了 string，所以到最后一行时，剩余可能性只能是 number。这叫控制流分析（control flow analysis）。

位置编译器眼中的 value为什么
进入函数number | string两种都可能
typeof === "string" 分支内string守卫保留了字符串
分支返回后的下一行numberstring 路径已结束

权威来源：TypeScript Handbook · Narrowing。本课示例均假设开启 strict。

2 · typeof 类型守卫：先区分 JavaScript 基本类型

typeof value 会在运行时返回一个字符串。TypeScript 认识这些比较，并据此收窄：

守卫分支内类型常见用途
typeof x === "string"string字符串方法、长度
typeof x === "number"number算术、toFixed
typeof x === "boolean"boolean布尔分支
typeof x === "undefined"undefined缺失值检查
typeof x === "object"对象，但也可能是 null必须继续排除 null

JavaScript 还可能返回 "bigint"、"symbol" 和 "function"。关键不是背表，而是记住：typeof 是运行时检查，TS 借它理解当前分支。

例 1：一层层剥开联合类型

type Input = string | number | boolean;

function describe(input: Input): string {
  if (typeof input === "string") {
    return `文本，共 ${input.length} 个字符`;
  }
  if (typeof input === "number") {
    return `数字，保留两位：${input.toFixed(2)}`;
  }
  return input ? "已开启" : "已关闭"; // 剩下 boolean
}

第二个 if 不是重新从三种类型开始：第一条路径返回后，此处起始类型已经只剩 number | boolean。收窄会沿着控制流持续生效。

常见坑 ①：typeof null === "object"

这是 JavaScript 的历史行为：typeof null 的结果也是 "object"。因此只检查 typeof value === "object"，并不能保证值真的可以读取属性。

type Profile = { name: string };

function printName(value: Profile | null) {
  if (typeof value === "object") {
    value.name; // ❌ value 仍可能是 null
  }

  if (typeof value === "object" && value !== null) {
    value.name; // ✅ value: Profile
  }
}

固定写法：检查对象时同时写 value !== null。数组的 typeof 也是 "object"；若要专门识别数组，应使用 Array.isArray(value)。

3 · 真值收窄 Truthiness Narrowing：先排除“假值”

JavaScript 把条件里的值转换成布尔值。false、0、""、null、undefined 和 NaN 都是假值（falsy）；其他值通常是真值（truthy）。TypeScript 能利用 if (x)、!x、&& 等表达式收窄。

type State = "ready" | "" | 0 | 1 | null | undefined;

function run(state: State) {
  if (state) {
    state; // 1 | "ready"：假值成员都被排除了
  } else {
    state; // "" | 0 | null | undefined
  }
}

字面量联合能精确表示 0 与 ""，所以收窄结果很精确。若原类型是宽泛的 string | null，进入 if 后会得到 string；但在 else 中不能只当成 null，因为宽泛的 string 仍包含空字符串。

常见坑 ②：0 和空字符串可能是合法数据

真值检查是在问“它是否为真”，不等于只问“它是否为空”。如果 0 或 "" 有业务意义，if (value) 会把它们一起挡掉。

function showScore(score: number | null) {
  if (score) {
    console.log(score); // 分数 0 不会进入这里！
  }

  if (score !== null) {
    console.log(score); // ✅ 0 也会显示
  }
}

选择原则：只有当所有假值都代表“没有数据”时才用真值收窄；否则明确比较 null 或 undefined。

4 · 相等收窄：用 === / !== 精确命中字面量

与一个字面量做严格相等或不等比较，会保留或排除对应成员。这比真值检查更精确，也更能表达业务含义。

type Command = "start" | "pause" | "stop";

function handle(command: Command) {
  if (command === "start") {
    command; // "start"
    return "开始运行";
  }

  command; // "pause" | "stop"，"start" 已被排除
  if (command !== "stop") {
    command; // "pause"
  }
}

例 2：分别处理 null、undefined 与有效值

function label(value: string | null | undefined): string {
  if (value === null) {
    return "已主动清空";
  }
  if (value === undefined) {
    return "尚未提供";
  }
  return value.toUpperCase(); // value: string，"" 仍合法
}

这里不能用一个 if (value) 替代，因为函数刻意区分了 null、undefined 和空字符串。严格相等检查让运行时语义与类型收窄完全对齐。

5 · in 操作符：按“有没有这个属性”区分对象

对象联合的成员拥有不同属性时，可用 "prop" in obj 检查属性是否存在。TypeScript 会把对象收窄到可能拥有该属性的成员。

type EmailContact = {
  email: string;
  sendEmail: () => void;
};

type SmsContact = {
  phone: string;
  sendSms: () => void;
};

function notify(contact: EmailContact | SmsContact) {
  if ("email" in contact) {
    contact.sendEmail(); // contact: EmailContact
  } else {
    contact.sendSms();   // contact: SmsContact
  }
}

in 右侧必须是对象。如果联合里还包含 null 或基本类型，要先用相等检查或 typeof 把它们排除。

可选属性会出现在两边

如果某个成员的属性是可选的，它既可能通过 in 检查，也可能不通过，因此不能被简单地放进某一边：

type Fish = { swim: () => void };
type Bird = { fly: () => void };
type Human = { swim?: () => void; fly?: () => void };

function move(animal: Fish | Bird | Human) {
  if ("swim" in animal) {
    animal; // Fish | Human
  } else {
    animal; // Bird | Human
  }
}

另外，JavaScript 的 in 也会检查原型链。它适合区分已知对象联合，但不要把它误当作完整的外部数据校验。

6 · instanceof：按“由哪个类创建”来收窄

value instanceof Constructor 会在运行时检查对象的原型链。TypeScript 因而能把值收窄为该类的实例。

function formatDate(value: Date | string): string {
  if (value instanceof Date) {
    return value.toISOString(); // value: Date
  }
  return value.trim();          // value: string
}

它尤其适合 Date、Error、DOM 类以及你自己定义的 class。与 typeof 相比，instanceof 能区分不同种类的对象。

常见坑 ③：interface 不能放在 instanceof 右边

interface 和 type 只存在于类型检查阶段，编译后会被擦除；instanceof 却需要一个运行时真实存在的构造函数。

interface User { name: string }

// value instanceof User
//                  ❌ User 只表示类型，运行时没有这个值

class Member {
  constructor(public name: string) {}
}

if (value instanceof Member) {
  value.name; // ✅ value: Member
}

7 · 重点：可辨识联合 Discriminated Unions

当联合的每个对象成员都有一个相同字段名，而该字段分别是不同的字面量类型时，这个字段就叫判别字段（discriminant），整个联合叫可辨识联合（discriminated union）。

最常用的字段名是 kind、type 或 status。它不是 TypeScript 的特殊语法，只是一种让数据结构主动携带“我是哪一种”的设计。

完整例子：Shape = Circle | Square

type Circle = {
  kind: "circle";
  radius: number;
};

type Square = {
  kind: "square";
  sideLength: number;
};

type Shape = Circle | Square;

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      // shape: Circle，radius 一定存在
      return Math.PI * shape.radius ** 2;

    case "square":
      // shape: Square，sideLength 一定存在
      return shape.sideLength ** 2;
  }
}

编译器看到 case "circle"，就知道 shape.kind 只能来自 Circle，因此可以安全读取 radius；另一个分支同理。

设计要件Shape 中的体现作用
统一字段名两个成员都有 kind提供统一判断入口
互不相同的字面量"circle" / "square"一对一识别成员
成员专属数据radius / sideLength收窄后安全访问

为什么不要把所有字段都做成可选？

type BadShape = {
  kind: "circle" | "square";
  radius?: number;
  sideLength?: number;
};

const broken: BadShape = {
  kind: "circle",
  sideLength: 10 // 居然合法：kind 与数据不一致
};

可选字段模型没有表达“circle 必须配 radius”这条约束。可辨识联合把每种状态拆成独立成员，能让不可能的状态无法构造，而不只是在读取时少报一个错。

8 · never 与穷尽检查：新增成员时别漏处理

当 switch 已排除联合里的每个成员，剩余值应当“不可能存在”。TypeScript 用 never 表示这种不可能类型。我们可以写一个 assertNever，让编译器替我们检查分支是否穷尽（exhaustiveness check）：

function assertNever(value: never): never {
  throw new Error(`未处理的形状：${JSON.stringify(value)}`);
}

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "square":
      return shape.sideLength ** 2;
    default:
      return assertNever(shape); // shape 应当是 never
  }
}

它如何在需求变化时保护你？

以后给联合新增 Triangle，却忘了给 area 增加分支：

type Triangle = {
  kind: "triangle";
  base: number;
  height: number;
};

type Shape = Circle | Square | Triangle;
// assertNever(shape) 处报错：
// ❌ Triangle 不能传给 never

这个错误正是在提醒：“你漏处理了 triangle。”never 本身会在 0007 深入学习；本课先记住它在可辨识联合中的一个实用角色：把未来新增成员造成的遗漏变成编译错误。

10 · 动手 B：组合守卫与提前返回

真实代码常会组合多次收窄。读下面例子，观察每次 return 如何永久排除一条路径：

function normalize(input: string | number | null): string {
  if (input === null) {
    return "—";
  }
  // input: string | number

  if (typeof input === "number") {
    return input.toFixed(2);
  }
  // input: string

  return input.trim();
}

这种“先处理特殊情况并返回”的写法叫提前返回（early return）。它减少嵌套，也让剩余路径的类型自然变窄。

11 · 如何选择守卫：从你真正想问的问题出发

你想知道什么？优先选择典型形式
是哪一种基本类型？typeoftypeof x === "string"
所有假值都可视为“没有”？真值收窄if (x)
是否等于某个状态或空值？相等收窄x === null、x !== "stop"
对象是否拥有区分性属性？in"email" in contact
是否由某个运行时类创建？instanceofx instanceof Date
业务对象属于哪个封闭变体？可辨识联合switch (x.kind)

最重要的判断：守卫既要让编译器满意，也要与业务语义一致。若 0 是合法分数，就不要用真值检查；若对象本来就是你设计的封闭状态集合，就优先加入判别字段，而不是到处猜测有哪些属性。
