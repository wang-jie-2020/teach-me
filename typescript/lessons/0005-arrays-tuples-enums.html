数组、元组与枚举

本课小胜利:能熟练写出"装同一类东西的列表"(数组)、"位置和类型都固定的小结构"(元组 tuple),并理解 enum 枚举 与 字面量联合类型 的取舍——这是写"形状更复杂"代码的拐点。约 22 分钟。

1 · 数组类型:两种写法完全等价

TypeScript 给"一组同类元素"提供了两种语法,完全等价,任选一种用就好(同一份代码里风格保持一致)。

// 写法 A:元素类型后接方括号(更短,推荐)
let scores: number[] = [95, 87, 76];

// 写法 B:泛型 Array<T>(在复杂类型嵌套时更易读)
let names: Array<string> = ["Ada", "Bob"];

类型推断会自然给出数组类型——声明即初始化时,不必手写:

let tags = ["ts", "js"];  // 推断: string[]
let nums = [1, 2, 3];        // 推断: number[]

数组方法的参数和返回值也都被自动类型化:tags.map(s => s.toUpperCase()) 里 s 就是 string。

来源:Handbook · Arrays。

2 · 只读数组:承诺"我不改它"

当一个数组应当只读(例如配置项、来自外部的固定数据),用 readonly 修饰:

function total(items: readonly number[]): number {
  return items.reduce((s, n) => s + n, 0);
}
total([1, 2, 3]);  // ✅ 读可以
items.push(4);     // ❌ Property 'push' does not exist on type 'readonly number[]'

同样有泛型写法:ReadonlyArray<T>,与 readonly T[] 等价。两者都不允许 push/pop/splice/sort/... 等任何会改变长度或顺序的方法。

const ro: ReadonlyArray<string> = ["a", "b"];
ro[0] = "x";   // ❌ 下标写入也被禁
ro.forEach(s => console.log(s.toUpperCase()));  // ✅ 读操作照常用

来源:Handbook · Arrays(同节)。

3 · 元组 tuple:长度固定、每位类型已知

数组表达"一堆同类",元组表达"一个固定形状的复合值"——典型场景:CSV 的一行、地图的 [经度, 纬度]、函数的 [状态码, 消息]。

// [经度, 纬度] —— 类型、顺序、长度都已知
let coord: [number, number] = [121.47, 31.23];

// [状态码, 消息]
type Result = [number, string];
const r: Result = [200, "OK"];

解构时,每个位置自动得到那个位置的类型:

const [lng, lat] = coord;
lng.toFixed(2);  // ✅ number 的方法
lat.toUpperCase(); // ❌ number 上没有 toUpperCase

3.1 可选元素与具名元素

用 ? 标记末尾可选:不写就少一位,写了就多位,中间不能空着。

// RGB 或 RGBA
type Color = [number, number, number, number?];
const c1: Color = [255, 0, 0];
const c2: Color = [255, 0, 0, 0.5];
const c3: Color = [255];   // ❌ 长度不够

从 TS 4.0 起,元组元素可以起名,只影响编辑器提示和可读性,不影响类型:

type Point = [x: number, y: number];
const p: Point = [10, 20];
p[0];  // 悬浮提示: (parameter) x: number

来源:Handbook · Object Types · Tuple Types。

4 · rest 元组:开头固定 + 末尾可变长

当"至少一个,后面跟若干同类"时,用 ...T[] 描述尾部:

// 一个 string 后面跟任意个 number
type Header = [string, ...number[]];

const a: Header = ["scores", 95, 87, 76];
const b: Header = ["scores"];          // ✅ 末尾 rest 允许 0 个
const c: Header = ["scores", "x"];  // ❌ 第二位必须是 number

典型用法——把 JS 的 concat/arguments 之类的不定参数包成精确类型。

5 · as const:把数组/对象"凝固"成只读元组

回忆 0002 讲过的 as const:它把所有字段从"宽类型"(string、number)收窄成"字面量类型",并把数组变成只读元组。

// 不加 as const: 推断为 string[]
const dirs = ["up", "down", "left", "right"];

// 加 as const: 推断为 readonly ["up", "down", "left", "right"]
const dirs2 = ["up", "down", "left", "right"] as const;

这给你两件事:只读 + 每个位置都是字面量类型(可以让 switch 写得既精确又穷尽)。

承接 0002;详见 Handbook · Literal Types。

动手 A:把"方向数组"变成"只读元组字面量"

6 · 枚举 enum:给一组命名常量一个"容器"

当你需要"几个互斥的具名值"(状态、方向、颜色代号),enum 是一种历史悠久、声明式的写法。

6.1 数字枚举(有"反向映射")

enum Direction {
  Up,      // 0
  Down,    // 1
  Left,    // 2
  Right,   // 3
}
let d: Direction = Direction.Left;  // 2

数字枚举有一个特殊性质——反向映射:从值也能查到名字。

Direction[2];        // "Left"  ← 编译产物里是对象 {Up:0, Down:1, ...} 加反向映射
Direction["Left"];  // ❌ 字符串枚举才能这么做

6.2 字符串枚举(更直观,无反向映射)

enum Color {
  Red   = "RED",
  Green = "GREEN",
  Blue  = "BLUE",
}
let c: Color = Color.Red;  // "RED"

字符串枚举的每一项必须显式赋值,运行时不创建反向映射,调试时看到的值更友好。

来源:Handbook · Enums。

7 · 重点对比:enum vs 字面量联合

在现代 TypeScript 里,多数情况下推荐用字面量联合替代 enum。下面两种写法在类型能力上几乎一样:

// —— 写法 A:enum ——
enum Dir { Up = "UP", Down = "DOWN", Left = "LEFT", Right = "RIGHT" }

// —— 写法 B:字面量联合 + 类型别名(更轻) ——
type Dir = "UP" | "DOWN" | "LEFT" | "RIGHT";

function turn(d: Dir) { /* ... */ }
turn("UP");  // ✅ 两种写法都接受

关键区别在"运行时"

维度enum字面量联合 "A"|"B"
运行时代码会生成一个真实的 JS 对象完全擦除,0 字节
值的使用Dir.Up(限定访问)直接写字面量 "UP"
反向映射数字枚举有,字符串枚举无无
tree-shaking对象整体保留,通常更难摇掉天然友好
类型能力强(可做 namespace、常量等)强(完全够用)

选型建议(经验法则):
• 仅在类型层用的有限集合 → 优先 字面量联合;
• 需要运行时枚举对象(例如遍历、反射)、或希望"访问时带命名空间" → 用 enum;
• 想要"既像 enum 又要 tree-shaking 友好" → as const 对象 + typeof X[keyof typeof X](见下)。

折中写法:as const 对象

const Dir = {
  Up:    "UP",
  Down:  "DOWN",
  Left:  "LEFT",
  Right: "RIGHT",
} as const;

type Dir = typeof Dir[keyof typeof Dir];  // "UP"|"DOWN"|"LEFT"|"RIGHT"

function turn(d: Dir) { /* ... */ }
turn(Dir.Up);   // ✅ 命名访问
turn("UP");     // ✅ 也接受字面量

常见坑 ①:enum 生成运行时代码(违反 0001 讲的"type erasure")

回想 0001:类型在编译时被擦除。enum 是个例外——它会生成一个真实的对象,出现在 JS 产物里。生产环境建议开启 "isolatedModules": true + "verbatimModuleSyntax",并谨慎评估是否真的需要它。

// TS 源码
enum E { A, B }
console.log(E.A);

// 编译后 JS(简化)
var E;
(function (E) {
    E[E["A"] = 0] = "A";  // 数字枚举的反向映射
    E[E["B"] = 1] = "B";
})(E || (E = {}));
console.log(E.A);

常见坑 ②:数字枚举的"隐式值"会让人误解

数字枚举默认从 0 自增。插入新成员或重排顺序会改值,而使用方拿到的是数字而不是名字——序列化、网络传输里会出人命。

enum Status { Pending, Running, Done }
JSON.stringify({ s: Status.Running });  // {"s":1} —— 你以为存的是 "Running" 吗?

经验:在需要序列化/跨网络/跨语言传递时,优先用字符串枚举或字面量联合,别用数字枚举。

9 · 小结与下一步

  • ✅ 数组:T[] 与 Array<T> 等价;readonly T[] / ReadonlyArray<T> 只读。
  • ✅ 元组:固定长度、位置类型;可选元素 ?、具名元素、rest 元素 ...T[]。
  • ✅ as const:把数组"凝固"成只读字面量元组,接 0002。
  • ✅ 枚举:数字枚举有反向映射、字符串枚举更直观;现代 TS 常优先字面量联合或 as const 对象。
  • ✅ 常见坑:enum 会生成运行时对象(违反 type erasure);数字枚举默认值的"重排灾难"。
