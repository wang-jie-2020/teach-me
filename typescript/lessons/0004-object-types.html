对象类型深入

本课小胜利：能为真实数据设计清晰的对象形状，正确处理可选与只读属性，读懂索引签名和多余属性检查，并用嵌套、interface 与交叉类型组合模型。约 28 分钟。

1 · 对象类型描述的是“形状”

JavaScript 中大量数据都以对象存在；TypeScript 的对象类型（object type）描述对象必须有哪些属性，以及每个属性允许什么值。它关心的是形状，而不是对象由哪个构造器创建。

function showUser(user: { name: string; age: number }): string {
  return `${user.name}（${user.age} 岁）`;
}

showUser({ name: "Ada", age: 36 });           // ✅ 形状匹配
showUser({ name: "Ada" });                    // ❌ 缺少 age
showUser({ name: "Ada", age: "36" });         // ❌ age 应为 number

这种“只要形状兼容就可以赋值”的规则叫结构类型（structural typing）。后面看似奇怪的“多余属性检查”，正是在这个基础规则之上额外加的一层防错机制。

本课程继续假设 tsconfig.json 中开启 "strict": true。主要来源：Handbook · Object Types。

2 · 同一个形状的三种写法

写法一：内联匿名对象类型

内联匿名（inline anonymous）写法把类型直接放在使用处，适合只出现一次、很短的形状。

// 例 1：类型紧挨参数，一眼就能看到契约
function move(point: { x: number; y: number }): number {
  return point.x + point.y;
}

若同一形状反复出现，继续内联会重复；这时应给它一个名字。

写法二：type 类型别名

类型别名（type alias）用 type 为一个类型命名。它不仅能表示对象，也能表示联合类型、交叉类型等。

// 例 2：命名后可在参数、变量、返回值中复用
type Point = {
  x: number;
  y: number;
};

function move(point: Point): number {
  return point.x + point.y;
}

写法三：interface 接口

接口（interface）也能为对象形状命名，成员写法与 type 几乎相同。

// 例 3：当前用法与上面的 Point 等价
interface Point {
  x: number;
  y: number;
}

const origin: Point = { x: 0, y: 0 };

此刻不必纠结二选一：表达普通对象形状时，两者都很好；遵循项目现有风格即可。它们的基本差异会在第 8 节点到。

3 · 可选属性 ?：可以缺席，不等于随时安全

属性名后的 ? 表示可选属性（optional property）：创建对象时可以不提供它。

// 例 4：nickname 可以存在，也可以缺席
type User = {
  id: number;
  name: string;
  nickname?: string;
};

const ada: User = { id: 1, name: "Ada" };       // ✅ 可省略 nickname
const lin: User = { id: 2, name: "Lin", nickname: "L" };

但读取 user.nickname 时，它的类型是 string | undefined：属性缺席时，JavaScript 读出的就是 undefined。因此不能直接当 string 使用。

function label(user: User): string {
  user.nickname.toUpperCase();
  // ❌ “user.nickname”可能为 undefined

  if (user.nickname !== undefined) {
    return user.nickname.toUpperCase(); // ✅ 判空后缩窄为 string
  }
  return user.name.toUpperCase();
}

短表达式可用可选链（optional chaining）和空值合并：

const nickname = user.nickname?.toUpperCase() ?? "未设置";

nickname?: string 与 nickname: string | undefined 不完全相同：前者允许属性整个缺席；后者要求属性必须存在，只是它的值可以为 undefined。

常见坑 ①：写了 ?，读取时却忘了判空

? 放宽的是“创建对象时是否必须提供”，不是承诺“使用时一定有值”。不要用 user.nickname! 随手压掉错误；只有当你掌握编译器不知道的可靠事实时，非空断言 ! 才合理。

4 · 只读属性 readonly：编译期防止重新赋值

属性名前加 readonly，表示只读属性（readonly property）。对象创建后，TypeScript 不允许通过该属性重新赋值。

// 例 5：订单编号一旦创建便不应改变
interface Order {
  readonly id: string;
  status: "pending" | "paid";
}

const order: Order = { id: "A-001", status: "pending" };
order.status = "paid";  // ✅ 普通属性可修改
order.id = "A-002";     // ❌ 无法分配到 id，因为它是只读属性

readonly 是编译期约束，不会在生成的 JavaScript 中冻结对象。它还是浅层的：只禁止替换属性本身，不会自动把属性内部的对象或数组也变只读。

interface Project {
  readonly tags: string[];
}

const project: Project = { tags: ["ts"] };
project.tags = [];       // ❌ 不能替换 tags
project.tags.push("web"); // ✅ 数组本身仍然可变

若连数组元素也不应改，可把属性写成 readonly tags: readonly string[]。记住：它提供类型检查，不等同于运行时的 Object.freeze()。

5 · 索引签名：描述“键名未知，值类型已知”

有些对象的属性名要到运行时才知道，例如各科成绩、商品库存。这时用索引签名（index signature）：

// 例 6：任意 string 键，都必须对应 number 值
type Scores = {
  [key: string]: number;
};

const scores: Scores = {
  math: 95,
  history: 88,
};

scores.physics = 91;       // ✅ number
scores.english = "A";     // ❌ string 不能赋给 number
const subject = "math";
const result = scores[subject]; // 动态键用方括号读取

key 只是参数名，可换成 subject 等更有语义的名字；真正的契约是“所有字符串键的值都是 number”。因此，同一类型中明确写出的属性也必须服从索引签名：

interface BrokenScores {
  [key: string]: number;
  total: number; // ✅ 符合索引签名
  name: string;  // ❌ string 属性不符合 number 值类型
}

更严格的动态读取：默认情况下，scores["不存在"] 的静态类型仍是 number，但运行时会得到 undefined。项目开启 noUncheckedIndexedAccess 后，这类读取会变成 number | undefined，迫使你判空。

练习：补全索引签名，使每个商品编号都映射到库存数量。

type Stock = {
  [productId: string]: ____;
};

6 · 多余属性检查：为什么直接写会更严格

先看最容易让初学者困惑的一组代码：

// 例 7：目标类型只要求 name 和 age
interface Profile {
  name: string;
  age: number;
}

function saveProfile(profile: Profile): void {}

saveProfile({ name: "Ada", age: 36, admin: true });
// ❌ 对象字面量只能指定已知属性，admin 不在 Profile 中

const candidate = { name: "Ada", age: 36, admin: true };
saveProfile(candidate); // ✅ 变量中转后，结构兼容：所需属性都存在

这不是前后矛盾，而是两条规则叠加：

  1. 一般赋值规则是结构兼容：来源对象至少拥有目标类型要求的属性即可，多一些通常没问题。
  2. 新鲜对象字面量有额外检查：对象字面量直接赋给已知类型、或直接作为参数传入时，TypeScript 会执行多余属性检查（excess property checks），专门捕捉拼写错误和误传字段。
const bad: Profile = { name: "Ada", age: 36, agge: 36 };
//                                                   ^ ❌ 很可能是 age 的拼写错误

所以“变量中转”确实会绕过这项额外检查，但不会删除运行时的 admin，也不应把它当成隐藏建模问题的技巧。若额外键本来就是需求，应在类型里明确表达。

需要开放额外键时：使用索引签名

// 例 8：name 必须是 string，同时允许其他未知属性
interface FlexibleProfile {
  name: string;
  [key: string]: unknown;
}

const profile: FlexibleProfile = {
  name: "Ada",
  age: 36,
  admin: true,
}; // ✅ 额外键的值先按 unknown 安全接收

使用 unknown 意味着读取额外字段后仍要缩窄类型，比 any 安全。如果键集合其实是固定的，就应明确列出属性，而不是加一个过宽的索引签名。

常见坑 ②：以为 TypeScript 使用“精确对象类型”

Profile 的含义不是“只能有 name 和 age”，而是“至少满足这些成员”。多余属性检查只针对新鲜对象字面量，目的是尽早抓住可疑字段；变量中转回到普通的结构兼容规则，所以行为才会“看起来奇怪”。

7 · 嵌套对象：从外到内逐层描述

嵌套对象类型（nested object type）没有新语法：某个属性的类型本身继续写成对象类型即可。复杂模型建议把有独立含义的层级拆成命名类型。

// 例 9：地址可选，订单项是对象数组
type Address = {
  city: string;
  postalCode: string;
};

type OrderItem = {
  sku: string;
  quantity: number;
};

interface CustomerOrder {
  id: string;
  customer: {
    name: string;
    address?: Address;
  };
  items: OrderItem[];
}

const order: CustomerOrder = {
  id: "A-001",
  customer: { name: "Lin" },
  items: [{ sku: "TS-BOOK", quantity: 2 }],
};

校验会逐层发生：items 中每个元素都必须同时拥有 sku 与 quantity。访问可选的 address 时，也要在对应层级处理 undefined：

const city = order.customer.address?.city ?? "城市未知";

练习：address 是可选属性。补上一个运算符，让地址缺席时不会直接访问 city。

const city = order.customer.address____city;

8 · interface、type 与交叉类型 &

阶段一先掌握最基本的选择依据；继承、联合与高级组合会在阶段二深入。

能力interfacetype
描述对象形状可以可以
扩展已有对象可用 extends可用交叉类型 &
同名声明合并可以，成员会合并不可以重复声明同名别名
直接表示联合类型不可以可以，如 type Id = string | number
// interface：extends + 声明合并
interface Person {
  name: string;
}
interface Employee extends Person {
  employeeId: number;
}

interface WindowOptions { width: number; }
interface WindowOptions { height: number; }
// WindowOptions 最终同时拥有 width 和 height

// type：可以给对象之外的类型命名
type Id = string | number;

阶段一经验法则：以对象为主、希望允许扩展或声明合并时可选 interface；需要联合或交叉组合时选 type。普通对象两者都能胜任，优先保持项目一致，而不是争论“唯一正确答案”。来源：Handbook · Everyday Types。

交叉类型：要求同时满足多个类型

交叉类型（intersection type）用 & 组合类型。对对象而言，结果需要同时拥有两边要求的全部成员。

// 例 10：组合业务数据和审计字段
type Article = {
  title: string;
  body: string;
};

type Timestamped = {
  createdAt: Date;
  updatedAt: Date;
};

type SavedArticle = Article & Timestamped;

const article: SavedArticle = {
  title: "对象类型",
  body: "……",
  createdAt: new Date(),
  updatedAt: new Date(),
};

& 不是简单地“后者覆盖前者”。若同名属性要求冲突，例如 { id: string } & { id: number }，那么 id 必须同时是 string 和 number，实际会落到无法正常提供的 never。组合前应确保同名成员兼容。

9 · 综合练习：把规则串起来

阅读类型并判断报错原因：

interface Config {
  readonly appName: string;
  theme?: "light" | "dark";
  limits: { [key: string]: number };
}

const config: Config = {
  appName: "Teach Me",
  limits: { uploads: 10 },
};

config.appName = "New Name";        // ① 报错：readonly
config.theme.toUpperCase();          // ② 报错：可能为 undefined
config.limits.users = "100";           // ③ 报错：索引值必须为 number

要修复第 ③ 处，"100" 应改成什么基本类型的值？填类型词。
