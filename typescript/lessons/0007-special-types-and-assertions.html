1 · any:关闭类型检查的逃生舱

any 是 TS 类型系统的“万能插槽”——把所有类型检查关掉,让一个值可以被赋给任何变量、调用任何方法。 官方原话:“If you don't want TypeScript to check a particular variable, you can use any.” (Everyday Types)

let x: any;

x = 1;
x = "hello";
x.toUpperCase();   // 编译过!运行时如果 x 是 1 → TypeError
x.foo.bar.baz;     // 编译过!运行时大概率 undefined

可以看到 any 实际上把你“退回到了纯 JS 模式”:编译器不再为你把关。

常见坑 ①:为什么尽量别用 any

每用一次 any,等于在你这一小片代码里关掉了 TS 的全部价值。 它会传染:从 any 出发访问到的成员默认也是 any,污染一片。 现实里几乎所有 any 都来自三类:

  • ① 第三方库没提供类型 → 改用 unknown + 收窄,或装 @types/xxx;
  • ② 数据结构还没想清楚 → 先用 unknown 顶住,后面再精确化;
  • ③ strict: false 偷偷漏出的隐式 any → 打开 strict 才是治本。

经验法则:any 只在“迁移老 JS、临时绕开类型 bug、且已加注释解释原因”时使用,否则默认禁止。

2 · unknown:安全的 any

unknown 表示“我也不知道这值是什么”,但它不会传染——必须先用类型守卫收窄(narrow)才能用。 这是 0004 narrowing 的直接应用。

function parse(input: string): unknown {
  return JSON.parse(input);   // 编译期不知道返回什么形状
}

const data = parse('{"n":42}');
console.log(data.n);          // ❌ Object is of type 'unknown'

// 必须先收窄
if (data && typeof data === "object" && "n" in data) {
  console.log((data as { n: number }).n); // ✅
}

对比 any vs unknown:

维度anyunknown
能赋给任何变量✅ 能✅ 能
能不检查直接访问属性✅ 能(危险)❌ 必须先收窄
传染性会传染整片代码不传染
何时用几乎不用 / 临时逃生“我不知道这是啥,我先顶住”

来源:Everyday Types。

3 · never:不可能的类型 + 穷尽检查

never 是“不可能的类型”—— 没有任何值可以属于它。它出现在两个场景:

  1. ① 永不返回的函数:抛异常、while(true){}。调用方拿不到返回值。
  2. ② 穷尽检查(exhaustiveness):把可辨识联合(0003)的所有分支都处理完后,剩余的“理论上不可能存在”的值类型就是 never。利用这一点写 assertNever。
function fail(msg: string): never {
  throw new Error(msg);
}

// —— 经典的 assertNever:给 union 加一个穷尽检查保护 ——
type Shape =
  | { kind: "circle";  r: number }
  | { kind: "square";  s: number }
  | { kind: "triangle"; base: number; h: number };

function assertNever(x: never): never {
  throw new Error("Unexpected: " + JSON.stringify(x));
}

function area(s: Shape): number {
  switch (s.kind) {
    case "circle":   return Math.PI * s.r ** 2;
    case "square":   return s.s * s.s;
    case "triangle": return (s.base * s.h) / 2;
    default:       return assertNever(s); // 加新分支没处理时这里报错
  }
}

如果以后给 Shape 加了 { kind: "trapezoid" } 而忘了在 switch 里写对应 case,assertNever(s) 会立刻在编译期报错——把“漏处理分支”挡在运行之前。 思路呼应 0003 的可辨识联合,工具是 0004 的 narrowing 收窄到 never。 来源:Narrowing。

为什么 never 能在 default 出现?

在 switch 把 circle / square / triangle 三个 case 都 return 之后, TS 知道剩下能走到 default 的 s 必然不属于 Shape——这种“剩余类型”就是 never。 而 assertNever 的形参正好声明为 never,编译器顺势校验:确实传进来的是 never,代码合法;如果哪天漏掉一个分支,这里类型不再为 never,编译就报错。 这把“忘记处理分支”这种最常见的运行时 bug 前置成了编译错误。

4 · void 与 undefined:别混淆

这是 0006 函数类型的延伸。一个返回 void 的函数表示“调用者不该关心返回值”,而返回 undefined 表示“真的返回了 undefined 这个值”。 来源:Functions · void。

function log(msg: string): void {
  console.log(msg);
  // 没有 return,或 return;
}

function makeNothing(): undefined {
  return undefined;   // 必须显式 return undefined
}

const a = log("hi");        // a: void
const b = makeNothing();    // b: undefined

常见坑:把 void 当成“返回了 undefined”——技术上没错,但语义不同。 void 是给调用方看的契约:“别用我的返回值”;undefined 是具体的值。 在回调签名里最常区分二者:回调参数写 () => void 容忍实现里意外返回点什么(常见于链式 API)。

5 · 类型断言 as 与 as const、非空断言 !

类型断言(type assertion)是告诉编译器“你比我懂,相信我”的语法。 它不是类型转换——不会改运行时行为,只是让编译期把类型按你说的算。 来源:Everyday Types · Type Assertions。

const el = document.getElementById("app") as HTMLDivElement;
// 运行时 el 仍然可能为 null;只是 TS 相信你,不再标红

断言 ≠ 转换。下面这段编译过、运行炸:

const x = "hello" as number;
console.log(x.toFixed(2));   // ❌ 编译过,运行报 x.toFixed is not a function

as const:把字面量“冻”成最窄类型

不加 as const,数组/对象的字面量会被推断成宽类型(如 string[]);加上后变字面量元组/只读对象(呼应 0002 的字面量类型):

const directions = ["up", "down", "left", "right"];
// 推断:string[]

const dirs = ["up", "down"] as const;
// 推断:readonly ["up", "down"]

type Dir = typeof dirs[number];  // "up" | "down"

这正是 0002 字面量 + 0003 union 的工厂组合:用 as const 收窄,再用 typeof 反推 union。

非空断言 !:对“一定不是 null”拍胸脯

const el = document.getElementById("app")!;
el.innerText = "hi";   // el 类型: HTMLElement(去掉了 null)

和 as 一样,只是关闭编译期怀疑——运行时如果真的是 null,立刻炸。 经验法则:能用 narrowing(if (el) ...)就别用 !;只有 100% 确信(比如紧跟着 document.body.appendChild 才查询的同一元素)才用,并加注释。

8 · 阶段综合 C:as const + union + 查表函数(交错练习 3)

把 0002(字面量)+ 0003(union)+ 0005(对象表)+ 0006(函数返回)+ 本课 as const 串起来。

下面有一个“语言 → 问候语”的查表函数。问题是 GREETINGS 没有用 as const,导致 keyof typeof GREETINGS 不是期望的字面量 union。

const GREETINGS = {
  en: "Hello",
  ja: "こんにちは",
  zh: "你好",
};
type Lang = typeof GREETINGS[keyof typeof GREETINGS];
// 当前 Lang 推断成:string(宽了,没区分能力)

function greet(lang: Lang): string {
  return GREETINGS[lang];
}

greet("en"); // ❌ 实际报: "en" 不是 Lang,因为 Lang 是 string

最小修改一行让 Lang 变成 "Hello" | "こんにちは" | "你好"? 填“加在 GREETINGS 后面的两个关键字”:

加上 as const 后,typeof GREETINGS[keyof typeof GREETINGS] 会按 union 求值,得到三种字面量值。这正是用“查表 + 字面量 + union”写出零开销、无 if/else 的纯查表函数的标准模式。
